## 1. P0 阻断 — SDD §7-10 主体内容未实际重写

### P0-1：SDD §7.1 / §7.4 / §7.5 / §7.6 / §9.1 / §9.2 / §9.3 实际章节仍为旧版本

**事实核查：**

- 修订草案 §0 修订摘要明确列出 7 个章节"修订要点"
- `git diff HEAD docs/spec/QuantPilot_SDD.md` 仅 18 行变更，**全部位于 §16 版本路线图**（行 1131~1153）
- 实际读取 SDD 行 332~620 区间，§7.1 仍是"横截面百分位排名法"，§7.4 仍是"V1.0 仅监控展示不自动调权"，§7.5 仍是固定 3×4 矩阵，§7.6 仍是 `Score = Σ(wᵢ × strategy_scoreᵢ) 0~100`，§9.1 仍是"评分 > 80"，§9.2 仍是"评分 < 40 / 浮亏 -8% / 软止损"，§9.3 仍是"得分构成字面陈述"
- system_design §9 Phase 15 行写明 "文档同步（SDD §7-10 合入 v1.3 草案）" —— 即**计划**把 §7-10 重写**推迟到 Phase 15 收尾**

**风险：**

SDD 是项目"权威单一来源"（CLAUDE.md §1 显式声明"设计文档与 SDD 冲突时，以 SDD 为准"）。当前状态下：

- 工程师在 Phase 11 实施期间打开 §7.1，看到"横截面百分位排名法"，可能按旧规范实现；
- 即便看到 §16 V1.0 收尾升级注，§16 的描述高度凝练（"Winsorize + 中性化 + Z-score + Gram-Schmidt + ICIR + 分位阈值 + 双重失效止损"），缺乏 §7.1~§9.3 级别的字段定义 / 数据流 / 边界条件；
- §7.2 四策略业务定义（趋势 / 动量 / 均值回归 / 价值）保持不变 —— 但 §7.2.1 趋势策略新增"共线性研究"附注（修订草案 §2.2 要求）在 SDD 中也**未追加**。

**该风险已经在 SDD 内部产生硬冲突**：
- SDD §7.4 行 440：「**V1.0 行为：** 仅监控并展示，不自动调权。」
- SDD §16 V1.0 收尾升级注（行 1136）：「评分引擎工业化（Phase 11）：…… **ICIR 滚动加权** ……」

同一份文档两个章节直接对立，且 §7.4 是 V1.0 范围内被"权威"标记的实施行为。

**修复方案（按优先级）：**

| 方案 | 工作量 | 一致性收益 | 推荐 |
|---|---|---|---|
| **A. 立即把修订草案 §2 全部合入 SDD §7-10**（替换旧章节，修订历史加一条 `v1.4`）| 中 | 最高，单一来源恢复 | ★ 推荐 |
| **B. 在 SDD §7 章首（行 330 与 332 之间）追加"⚠️ 重要修订说明"段**：指明 §7.1 / §7.4 / §7.5 / §7.6 / §9.1 / §9.2 / §9.3 七章节自 2026-05-14 起以 `sdd_7_10_revision_draft_2026-05-14.md` v1.3 为准，本主文档将在 Phase 15 正式合入；同时**在 §7.4 行 440 删除或改写 "V1.0 行为：仅监控并展示，不自动调权"** | 小 | 中（双源仍存在但冲突消解）| 最低成本兜底 |
| **C. 维持现状仅 §16 备注** | 0 | 低，冲突仍存在 | 不接受 |

**建议：** 启动 Phase 11 设计文档**之前**至少做 **方案 B**（30 分钟工作量）；如果 Phase 11 设计文档作者愿意先做 SDD §7-10 主体重写再写 Phase 11 详细设计，则做 **方案 A**（约 2-4 小时）。

## 2. P1 主要问题

### P1-1：SDD §7.4 V1.0 行为说明（行 440）与 §16 直接冲突

详见 P0-1 末段。**修复必须**：把 §7.4 行 440 的"V1.0 行为：仅监控并展示，不自动调权 / V1.5 行为：因子衰减严重时自动降低对应策略权重" 改写为：

> **V1.0-r6 行为（Phase 11 工业化后）**：每月末 ICIR 滚动校准 + 自动加权（详见 §7.4 修订草案 / `sdd_7_10_revision_draft_2026-05-14.md` v1.3）。原 V1.0 "仅监控展示"行为在 Phase 1~10 期间有效，Phase 11 完成后被 ICIR 自动加权机制取代。

否则在 Phase 11 工程师审阅 SDD §7 时会看到内部矛盾。

### P1-2：Phase 11 设计文档前置依据 vs 文件未创建

system_design §9 Phase 11 行末尾："**前置依据**：`docs/design/sdd_7_10_revision_draft_2026-05-14.md`（v1.3 已锁定）" —— OK，前置依据指向修订草案而非 Phase 11 设计文档自身，这是正确的。

但 system_design §9 表格中 Phase 11~15 的 "详细设计文档" 列分别填了：
- `phase11_scoring_industrialization.md`
- `phase12_factor_lineage.md`
- `phase13_production_observability.md`
- `phase14_account_integrity.md`
- `phase15_v1_rc_release.md`

**实际状态：** 五份 phase 设计文档**均不存在**（`docs/design/phases/phase1[1-5]*.md` 无匹配）。

按 CLAUDE.md §9 末 "**开始新 phase 前**：确认对应设计文档已存在于 `docs/design/phases/`，若不存在先创建"——这是显式约束。

**风险：** 当前 system_design 表格列出文件名但文件不存在，读者可能误以为已交付。

**建议：** 在 system_design §9 Phase 11~15 行的"详细设计文档"列暂时把文件名后加 `（待创建）`标记，或在表格下方注释"Phase 11~15 设计文档由各 Phase 启动时创建，当前尚未交付"；待 Phase 11 设计文档完成后去掉标记。

### P1-3：Phase 11~15 估算 ~38-55 pd 的拆分来源不明

CLAUDE.md §9 写 "估算 ~38-55 pd（8-11 周）"；roadmap §★ 升级清单末"小计：17 项升级 …… **~38-55 pd**（详见 system_design §9）"；但 system_design §9 Phase 11~15 行**没有 pd 估算列**。

**建议**（任选其一）：
- 在 system_design §9 Phase 11~15 各行末尾加上 pd 估算列，或在表格下方加"Phase 11 ~12-18 pd / Phase 12 ~6-10 pd / Phase 13 ~8-12 pd / Phase 14 ~3-5 pd / Phase 15 ~6-8 pd（合计 ~38-55 pd / 8-11 周）"
