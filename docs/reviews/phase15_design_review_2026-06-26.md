## 4. 问题清单

### F-1（P1）「§10.4 跨制度回归 / STRONG ≥30」出处错归 SDD/system_design

- **现象**：设计三处把 §10.4 归为 SDD / system_design 章节：
  - 头依据 line 7「SDD §10.4（跨制度回归）」— 实际 SDD §10.4 = **压力测试（V2.0）**。
  - 头依据 line 8「system_design … §10.4 跨制度回归基线」— 实际 system_design §10.4 = **可维护性**。
  - 15-3「**system_design §10.4** + SDD §9.1 … 把 STRONG 验收基线由绝对数「≥30」改为相对百分比」。
- **实证**：跨制度回归表与「各 trade_date ≥ 30 只 STRONG（宇宙 ~3200 × 1% ≈ 32）」基线**仅存在于 `phase11_scoring_industrialization.md §10.4`（line 945/951）**；system_design 全文无「≥30 STRONG / 跨制度回归」正文；SDD 侧相关数字在 **§9.1**（line 682「STRONG … 约前 30~35 只」+ line 688「理论 top 5% 候选 = 160~175 只」），非 §10.4。
- **影响**：15-3 是"文档校核/交叉一致性"核心交付，却把待修条款指向**不含该内容的 system_design §10.4 / SDD §10.4**。实施者按字面会改错章节或找不到目标，基线修正落空——对一个职责就是"消除文档不一致"的 phase，这是自相矛盾的硬伤。**定为 P1**。
- **修法**：
  1. 头依据 line 7/8 的「SDD §10.4 / system_design §10.4 跨制度回归」改为「**phase11 §10.4 跨制度回归基线**」（§10.4 是 phase11 设计文档章节号，非 SDD/系统设计）。
  2. 15-3 待修目标改为：**phase11 §10.4（line 951）`≥30 只` → 相对百分比** + **SDD §9.1（line 682「约前 30~35 只」、line 688「160~175 只」）口径同步**；若确需把验收基线提升进 system_design，则明确"新增 system_design 一节/一行"而非"改 §10.4"。
  3. §2/§3 正文凡"§10.4 跨制度回归"统一标注其文档归属，避免裸 §10.4 再被误读为 SDD/系统设计。
- **处置**：放行前修。

### F-2（P2）alembic 0017 与已批准的 V1.5-G 撞号

- **现象**：盘内最新迁移 0016（`0016_trade_fundflow_void.py`）；Phase 15 §8 占 **0017**（DROP factor_ic_history）；V1.5-G §3.3/§10/§11 已声明 **0017**（user 表 + account.user_id）。
- **影响**：Phase 15 先于 RC 落地、V1.5-G 排 RC 后 → 真实合入顺序 Phase 15 = 0017，V1.5-G 必为 0018+。两文档同号将来 `down_revision` 冲突 / 实施者混淆。Phase 15 占 0017 **正确**，但未提示 V1.5-G 须改号；V1.5-G 文档现已 stale。
- **修法**：Phase 15 §8 加一句「本 phase 先占 0017；V1.5-G §3.3 的 0017 须顺延 0018（已知会下游）」。鉴于 15-5 改名 grep 已会触碰 V1.5-G 文件，可顺带在该子项备注同改 V1.5-G alembic 号。或在 V1.5-G 文档侧改 0018。
- **处置**：放行前修（择一处落定，避免双 0017 进代码）。

### F-3（P2）15-1 阻断门槛「日均 BUY 5~15」与 phase11 实测「43~50 每 trade_date」未对账

- **现象**：15-1 信号链路验收阈值「日均 BUY 5~15」，且 §10「FAIL 即阻断 RC」。但 phase11 §10.4 收尾行（line 1012）5y 实测「BUY signals **43~50** 每 trade_date」（pool_capacity 20→50 后）。两者相差约 3 倍，无对账说明。
- **影响**：若生产日 BUY 真为 43~50，则 15-1 按 5~15 直接 FAIL → 误阻断 RC；若 5~15 才是目标，则需说明为何与 phase11 实测背离（回测 pool=50 上下文 vs 日常 pipeline 信号发射门槛差异？风控/持仓/watchlist 二次过滤？）。阻断门槛必须有据。
- **修法**：15-1 补"BUY 区间口径"说明：明确 5~15 是日常 pipeline 经 RiskChecker/仓位/watchlist 过滤后的可执行 BUY，与 phase11 回测期 pool 内 43~50 的差异归因；或据实把阈值改为与实测一致的区间。给出测量方法（哪条路径计数）。
- **处置**：放行前修（阻断门槛不可悬空）。

### F-4（P3）rollup 41-61 与分项和 40-62 略偏（~ 容差内，可选）

- **现象**：分项 Phase 11(12-18)+12(6-10)+13(8-12)+14(7-12)+15(7-10) = **40-62**；system_design 改为「~41-61」（按旧值 40-59 套 +1/+2 delta 而非重算）。旧值 40-59 本身亦非精确分项和（旧分项和 39-60）。
- **影响**：带「~」近似前缀，影响轻微；但与 F 系列前两次评审同类（估算传播）——上界 61 比分项和 62 少 1。
- **修法**：重算为「~40-62」或保留「~41-61」并注"区间为近似、非分项精确和"。可选，低优先。
- **处置**：折入实施核查 / 可不改。

### F-5（P3）15-5 roadmap 改名 grep 范围漏 `docs/reviews/`

- **现象**：15-5 改名 `v1_5_roadmap.md → v1_post_release_roadmap.md`，grep 替换范围列「SDD/system_design/各 phase 文档/CLAUDE.md/memory」，**未列 `docs/reviews/`**。但 V1.5-G 评审报告（`v1_5_g_multiuser_design_review_2026-06-26.md`）+ 本报告均引用旧名。
- **影响**：DoD 又说"仅剩历史修订记录中的旧名"——评审报告属历史归档，按惯例不重写。但归档报告里指向被改名文件的引用会成悬挂链接。
- **修法**：15-5 明确策略：①改名后在新文件头加"原名 v1_5_roadmap.md（2026-06-26 改名）"重定向注，归档 reviews/ 旧名引用保留为历史可接受；或②grep 范围显式纳入 reviews/。二选一写入 15-5。
- **处置**：折入实施核查。
