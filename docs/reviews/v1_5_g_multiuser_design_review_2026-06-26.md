## 0. 评审结论

**有条件通过（设计草案层面）。**

设计实质优良：per-user / shared 架构边界（§2）是全文最关键、也最正确的决策——准确切出"账户层隔离 / 市场计算层共享"的断层线，避免给每用户单独跑每日管线的算力灾难；与 system_design 共享计算层定位、SDD §2/§9.3/§14 的 L1/L2/L3 定义三向自洽。L1/L2/L3 定性为"自选偏好而非 RBAC"贴合 SDD §2 原文，且为后续 gating 设计扫清了"权限边界"误区。

但作为 **C-5 范围变更回写**，文档自我声明的回写范围（SDD §1/§2/§16 + system_design §3/§6/§9 + roadmap）**未全部兑现**，且 roadmap 总估算出现算术漂移。这些均为纯文档问题，不阻断设计成立，但在标记"C-5 回写完成 / 进入实施排期"前必须收口。

- 阻断级（P1）：**0**
- 重要（P2）：**2**（F-1 roadmap 总估算未同步、F-2 SDD §16 权威路线图未登记多用户）

放行条件：修 F-1 + F-2（纯文档），并把 F-3/F-4/F-5 写入本设计 §1.3 实施启动核查清单即可。

## 3. 设计亮点

1. **§2 架构边界表**：per-user / shared 二分清晰且穷举（逐表归类），并给出"派生语义"（共享信号 + 按用户账户叠加持仓视图），把最难的多用户改造收敛为"管线与账户解耦 + API 期实时叠加"，避免 per-user 管线。整合性最强的一节。
2. **L1/L2/L3 定性纠偏**：明确"偏好非 RBAC"，并实证当前 `user_config.user_level` 是"配置项最低层级"而非"用户层级"、根本无"用户的 level"承载主体——精准对接 SDD 推迟项的真实缺口。
3. **【设计待定】均带去向**：每日管线持仓保护（§2）、止损预警多用户化（§6.4）、邮箱验证（§4.3）、注册自动登录（§4.4）四项待定全部给出充分理由 + 三链承诺或推迟路由（V1.5-F），符合 C-3「现在的问题现在处理 / 推迟须落三链」。
4. **安全红线**：ownership 越权返 404 非 403（不泄露资源存在性）+ 登录沿用 verify_password 先行防计时侧信道（CLAUDE.md §7）+ 注册响应统一文案/耗时防枚举——安全意识到位。
5. **迁移幂等**：env admin → 首用户单事务 5 步 + 缺 env 中止不静默建无主账户（C-4 不静默掩盖）。

## 4. 问题清单

### F-1（P2）roadmap §6 总估算未随 V1.5-G 上调同步

- **现象**：V1.5-G 由 8-10 → 9-13 pd（+1~+3），但 §6 合计行 `~56-82 pd`、§1（line 47）、§6 标题（line 223）三处引用均未更新。
- **影响**：估算 rollup 算术漂移；与上次 phase14 评审 P2-1（system_design §9 累计估算脚注未同步）同类——估算变更未向上传播。
- **修法（纯文档，二选一）**：①三处改 `~57-85 pd`；②若总区间有意吸收增量，§6 合计行加一句"V1.5-G +1~3 pd 已吸收于区间上沿"。
- **处置**：放行前修。

### F-2（P2）SDD §16 版本路线图未登记 V1.5-G 多用户范围

- **现象**：设计 §1.3 C-5 行 + §10 DoD 声称回写 SDD §16，实际 §16 V1.5「用户设置」行无多用户/注册/层级登记。
- **影响**：C-5「SDD 是权威源」——权威路线图缺新范围，仅靠下属 roadmap 文档承载；同时构成"声称回写而未执行"的追溯不实。
- **修法（纯文档，二选一）**：①SDD §16 V1.5 表新增/扩写一行，如「用户与账户 | 开放自助注册 + 多用户账户隔离 + L1/L2/L3 自选偏好（user.level）」；②若有意保持 §16 高层不细化，则从设计 §1.3/§10 的回写声明中**移除 §16**，改为「§16 维持高层，细化登记在 roadmap」并说明。倾向①（保持 SDD 权威完整）。
- **处置**：放行前修。

### F-3（P3）SDD §1.1 缺前向注

- **现象**：DoD/§1.3 列「SDD §1」回写；当前仅 §2「实现状态」注间接提及"§1.1 边界扩展为多用户"，§1.1 正文「面向个人投资者」无前向引用。
- **影响**：轻微。§2 注已覆盖语义，但读 §1.1 者不会被导向 V1.5-G。
- **修法**：§1.1 末尾加一行「> V1.5-G 起边界扩展为多用户（开放注册 + 账户隔离），见 §2 注 / `v1_5_g_multiuser.md`」。或在实施启动核查兑现。
- **处置**：折入实施启动核查清单即可。

### F-4（P3）env admin 登录模型遗漏文档枚举不全

- **现象**：设计 §11 仅点名"更新部署文档"。实际 env admin 登录模型散落于：`docs/guides/deployment.md`、`docs/guides/dev_setup.md`、`CLAUDE.md` §4.6（override_admin_password fixture / 登录验证）+ §4.7（ADMIN_USERNAME/HASH 环境变量）、`.env.prod.example`、phase6/phase9 设计文档。
- **影响**：实施期若仅改部署文档，CLAUDE.md 项目宪法与开发指南将与"登录查 user 表"实现不一致。
- **修法**：本设计 §11 或 DoD 文档项展开枚举上述清单（尤其 CLAUDE.md §4.6/§4.7 与 `override_admin_password` autouse fixture 的多用户化改造）。
- **处置**：折入实施启动核查 / DoD 文档项。

### F-5（P3）SDD §14「适用层级」标签与 user_config.user_level 表示法映射

- **现象**：SDD §14 设置表「适用层级」列用 `All / L2+ / L3`；设计 §6.3 用 `user_config.user_level <= current_user.level`（L1/L2/L3 枚举）做字符串比较。`All`→L1、`L2+`→L2 的映射未在设计中明示；`'L2+'` 字面无法直接参与 `<=` 比较。
- **影响**：实施期 user_config 种子数据/迁移需把 §14 标签规范化为 L1/L2/L3 枚举值，否则过滤逻辑失效。
- **修法**：设计 §6.3 补一句映射规约（All=L1）；实施时校验 user_config.user_level 存量值均为 L1/L2/L3。
- **处置**：折入实施启动核查。
